## 1. Cel

US-27 rozszerza anulowanie (FR-24): CONFIRMED booking (opłacony) można anulować z **Stripe refundem**. Domyka też known limitation z US-25 (paid-after-cancel) — user może teraz anulować CONFIRMED i odzyskać pieniądze.

Zakres:

1. **`Booking.can_be_cancelled()`** — broaden `PENDING` → `PENDING | CONFIRMED` (+ reguła 1h bez zmian).
2. **`apps/payments/services.py::create_refund(booking) -> str`** — `stripe.Refund.create` (static idempotency key), zwraca refund id, raises StripeError.
3. **`cancel_booking`** — refund branch: CONFIRMED + payment_intent → `create_refund` **wewnątrz atomic** + zapis `refund_id`/`refunded_at` PRZED status change; Stripe error → rollback + `RefundError`.
4. **Testy** — broaden model + service refund (success/failure) + view CONFIRMED-cancel + `mock_refund` fixture.

### Decyzje (inline, per m3_planning — bez osobnego brainstormu)

| # | Decyzja | Wybór | Powód |
|---|---------|-------|-------|
| 1 | Refund vs atomic boundary | **Stripe Refund INSIDE `transaction.atomic()`** (przed status change) | Refund + CANCELLED muszą być atomic — nigdy CANCELLED-bez-refundu. Stripe error → rollback → status zostaje CONFIRMED. (≠ US-20 after-commit: tam brak money side-effect + high-contention lock; tu money + low-freq single-row lock) |
| 2 | Idempotency key | **Static `booking-{id}-refund`** (NIE per-call uuid) | Jeden refund per booking; retry tej samej operacji → Stripe zwraca ten sam refund (no double). (≠ checkout per-call uuid: tam chcemy fresh session per retry) |
| 3 | Refund failure | **Rollback + `RefundError(BookingError)`** → flash, status unchanged | already-refunded/card-revoked/network → "Anulowanie nieudane — skontaktuj się z obsługą"; brak CANCELLED-bez-money-back |
| 4 | can_be_cancelled broaden | **`status in (PENDING, CONFIRMED)`** | FR-24: CONFIRMED cancellable z refundem. PENDING nadal bez Stripe |

### Out of scope (defer'd)

- **Auto-refund w webhook** (paid-after-cancel z US-25) — user-initiated cancel pokrywa większość; auto-refund completed-dla-CANCELLED to osobny edge (poza M3).
- **Partial refunds** — full refund tylko (FR-24).
- **Admin** (BookingAdmin refund display) → US-28.

## 3. Model — `apps/booking/models.py::can_be_cancelled` (broaden)

```python
    def can_be_cancelled(self) -> bool:
        """True when this booking may still be cancelled by its owner (FR-10/FR-24).

        PENDING or CONFIRMED bookings up to 1h before the screening. CONFIRMED
        cancellation triggers a Stripe refund (US-27); PENDING cancels without Stripe.
        """
        return (
            self.status in (BookingStatus.PENDING, BookingStatus.CONFIRMED)
            and self.screening.start_time > timezone.now() + timedelta(hours=1)
        )
```

Zmiana: `== PENDING` → `in (PENDING, CONFIRMED)`. Reszta (1h rule) bez zmian.

## 4. Payments — `apps/payments/services.py::create_refund` (new)

```python
def create_refund(booking: Booking) -> str:
    """Refund a CONFIRMED booking's payment via Stripe; return the refund id.

    Static idempotency key (booking-<id>-refund) so a retry refunds at most once
    (unlike checkout's per-call key — there's only ever one refund per booking).
    Raises stripe.StripeError on failure (already-refunded, card-revoked, network).
    """
    refund = stripe.Refund.create(
        payment_intent=booking.stripe_payment_intent_id,
        idempotency_key=f"booking-{booking.id}-refund",
    )
    return refund.id
```

`stripe` + `Booking` już importowane w payments/services.py (US-24). `refund.id` to `str` (brak `str|None` jak `session.url`).

## 5. Service — `apps/booking/services.py` (extend)

Nowy wyjątek + refund branch w `cancel_booking`. Importy: `import stripe` (dla `except stripe.StripeError`) + `from apps.payments.services import create_refund` (obok istniejącego `create_checkout_session`).

```python
class RefundError(BookingError):
    """Stripe refund failed during cancellation — booking left unchanged."""

    def __init__(self) -> None:
        super().__init__(
            "Anulowanie nieudane — zwrot płatności nie powiódł się. Skontaktuj się z obsługą."
        )


def cancel_booking(*, booking: Booking) -> Booking:
    """Cancel a booking race-safely (FR-10 + FR-24 refund).

    Locks the booking, re-checks can_be_cancelled() under the lock. A CONFIRMED
    booking with a payment is refunded via Stripe INSIDE the transaction (static
    idempotency key) before the status flips — so we never end up CANCELLED without
    a refund; a Stripe failure rolls back and raises RefundError. PENDING bookings
    cancel without any Stripe call. Caller verifies ownership.
    """
    with transaction.atomic():
        locked = Booking.objects.select_for_update().get(pk=booking.pk)
        if not locked.can_be_cancelled():
            raise BookingNotCancellableError()
        if locked.status == BookingStatus.CONFIRMED and locked.stripe_payment_intent_id:
            try:
                refund_id = create_refund(locked)
            except stripe.StripeError as exc:
                raise RefundError() from exc
            locked.refund_id = refund_id
            locked.refunded_at = timezone.now()
        locked.status = BookingStatus.CANCELLED
        locked.expires_at = None
        locked.save(update_fields=["status", "expires_at", "refund_id", "refunded_at"])
    return locked
```

### Decyzje uzasadnione

1. **Refund INSIDE atomic, PRZED status change** (decyzja #1). Stripe error → exception → atomic rollback → status zostaje CONFIRMED. Success-then-DB-crash → retry: `create_refund` static key zwraca ten sam refund (no double) → save → CANCELLED.
2. **`except stripe.StripeError → raise RefundError() from exc`** — domain error (podklasa BookingError) złapany przez view → flash. `from exc` zachowuje traceback.
3. **Guard `CONFIRMED and stripe_payment_intent_id`** — refund tylko dla opłaconych. CONFIRMED bez payment_intent (nie powinno się zdarzyć) → cancel bez refundu (defensywne). PENDING → cancel bez Stripe.
4. **`save(update_fields=[..., "refund_id", "refunded_at"])`** — dla PENDING refund_id/refunded_at zostają default (""/None) — write no-op. OK.
5. **`import stripe` w booking/services.py** — potrzebne do `except stripe.StripeError`. (US-24 trzymało stripe w view; tu service łapie bo refund jest w service layer.)

## 8. Risks

1. **Refund-then-DB-crash.** Refund success ale commit fail → money back, booking CONFIRMED. Retry: static idempotency key → Stripe zwraca ten sam refund (no double) → CANCELLED. Eventually consistent. (Decyzja #1/#2.)
2. **Lock held across Stripe refund call.** `select_for_update` na Booking trzymany przez network call. Akceptowalne: cancel to low-freq single-row (≠ US-20 high-contention Screening lock). (Decyzja #1.)
3. **`import stripe` w booking/services.py + circular?** booking.services importuje payments.services (create_refund/create_checkout_session). payments.services importuje booking.models. Brak cyklu (models ≠ services). `import stripe` to external. OK.
4. **US-23/US-25 test regression od broaden.** `test_confirmed_false` (model), `test_raises_for_confirmed` (service), `test_not_cancellable_flashes_error` (view) zakładały CONFIRMED=not-cancellable. US-27 je aktualizuje (§6). Webhook tests (US-25) nie dotykają can_be_cancelled — bez wpływu.
5. **`mypy` `create_refund`.** `refund.id` to str (stripe Refund.id typed str). `cancel_booking` zwraca Booking. `except stripe.StripeError` — runtime class, no annotation. OK.
6. **CONFIRMED bez `stripe_payment_intent_id`.** Guard `and locked.stripe_payment_intent_id` → cancel bez refundu (defensywne; nie powinno się zdarzyć bo US-25 zawsze ustawia przy CONFIRMED). Brak crash.
